# Agent 三种协作模式：蜂群、Sub、链式

本文基于 [Agent 蜂群模式（Swarm）解析](https://juejin.cn/post/7603575399255949352)，解释 Heicode 中需要区分的三种 Agent 协作模式：

- 蜂群模式：多个 Agent 围绕共享环境自组织协作。
- Sub 模式：主 Agent 拆分任务，多个子 Agent 分工执行。
- 链式模式：多个 Agent 按固定顺序依次处理任务。

这三种模式解决的是“多个 Agent 如何协作”的问题。它们不是计费套餐，也不是模型等级。

## 一句话理解

| 模式 | 一句话 |
|---|---|
| 蜂群模式 | 没有强中心调度，多个 Agent 基于共享状态自主协作，最终收敛出结果。 |
| Sub 模式 | 有主 Agent 负责拆分、分配、汇总，子 Agent 按职责完成局部任务。 |
| 链式模式 | 任务按固定步骤从上一个 Agent 交给下一个 Agent，像流水线一样执行。 |

## 一、蜂群模式

蜂群模式对应文章中的 Swarm 思想。它的核心不是“多开几个 Agent”，而是让多个 Agent 围绕共享环境协作。

文章中提到，蜂群模式受自然界蚁群、蜂群启发：单个个体能力有限，但大量个体通过局部信息、间接通信和环境反馈，可以形成复杂的整体行为。

在 Agent 系统中，可以理解为：

```text
共享任务环境
-> 多个 Agent 感知局部状态
-> Agent 自主选择或生成任务
-> 更新共享状态
-> 其他 Agent 感知变化并继续协作
-> 达到收敛条件后输出结果
```

### 蜂群模式的关键特征

| 特征 | 说明 |
|---|---|
| 共享环境 | 所有 Agent 围绕任务池、共享状态、约束条件工作。 |
| 自组织 | Agent 不完全依赖一个中心调度器，而是根据局部信息行动。 |
| 隐式协作 | Agent 可以通过状态变化、事件、结果沉淀影响其他 Agent。 |
| Handoff | 某个 Agent 可以把任务、上下文或控制权交给另一个 Agent。 |
| 收敛判断 | 系统需要判断什么时候停止探索并输出结果。 |
| 鲁棒性 | 单个 Agent 失败时，其他 Agent 可以继续推进或补全。 |

### 蜂群模式适合什么任务

适合：

- 多源资料研究。
- 大规模代码审查。
- 复杂故障排查。
- 多方案探索。
- 安全、性能、测试多维度并行检查。
- 子任务会动态生成、互相影响的复杂任务。

不适合：

- 很小的单步任务。
- 预算非常紧的任务。
- 必须严格线性执行的任务。
- 强一致性事务。
- 没有明确停止条件的任务。
- 涉及生产高危操作但没有审批边界的任务。

### Heicode 中蜂群模式需要额外控制什么

蜂群模式能力强，但也更容易发散。因此在 Heicode 中，蜂群模式必须保留平台约束：

- 最大 Agent 数量。
- 最大任务轮次。
- 最大预算。
- 任务池边界。
- 共享状态规则。
- 收敛条件。
- 高危操作审批。
- 人工停止和接管。
- 审计记录。

这也是文章中的 Swarm 思想落地到真实客户环境时必须补上的部分。

## 二、Sub 模式

Sub 模式是 Heicode 当前更可控的多 Agent 协作方式。它不是完整蜂群模式，而是蜂群模式的基础能力。

Sub 模式的核心是“主 Agent 可控拆分”。主 Agent 负责理解目标、拆分任务、分配给子 Agent、接收结果并统一汇总。

```text
用户目标
-> 主 Agent 理解目标
-> 主 Agent 拆分子任务
-> 子 Agent 分工执行
-> 子 Agent 交付结果
-> 主 Agent 聚合和验证
-> 输出最终结果
```

### Sub 模式和文章中的 Supervisor 模式关系

文章中对比了 Swarm、Supervisor、Chain。Heicode 的 Sub 模式更接近“Supervisor + 子 Agent 分工”的形态：

- 有一个主 Agent 负责协调。
- 子 Agent 的职责由主 Agent 或用户明确指定。
- 子任务边界相对清楚。
- 结果由主 Agent 汇总。
- 风险、权限和预算更容易控制。

但 Heicode 的 Sub 模式不只是简单 Supervisor。它还会结合任务视图、权限边界、审批、用量和审计，让子 Agent 协作可以用于真实交付。

### Sub 模式适合什么任务

适合：

- 多模块开发。
- 复杂 bug 修复。
- 文档、前端、后端、测试需要分工的任务。
- 需要 Reviewer 独立审查的任务。
- 任务可以提前拆成几个明确部分的场景。

不适合：

- 子任务会不断动态生成的任务。
- 需要大量 Agent 自主竞争或补全的任务。
- 没有主线目标的开放探索。
- 更适合固定流水线的简单任务。

### Sub 模式的价值

- 比单 Agent 更能处理复杂任务。
- 比蜂群模式更容易控制风险。
- 可以给不同子 Agent 设置不同权限。
- 可以让执行、审查、部署分离。
- 可以按子任务查看进度、失败原因和模型用量。
- 适合作为蜂群模式上线前的基础能力。

## 三、链式模式

链式模式对应文章中的 Chain。它的核心是“固定顺序流水线”。

任务会按预设步骤依次传递：

```text
Agent A
-> Agent B
-> Agent C
-> Agent D
-> 输出结果
```

链式模式中，每个 Agent 只处理上游传来的内容，然后把结果交给下游。

### 链式模式适合什么任务

适合：

- 流程固定的任务。
- 数据清洗、分析、报告生成。
- 文档格式化。
- 固定发布检查。
- 固定审核流程。
- 不需要动态调整路径的任务。

不适合：

- 复杂探索。
- 多方案竞争。
- 任务路径不确定。
- 需要多个 Agent 并行补全。
- 上游结果不稳定且需要反复回退的任务。

### 链式模式的价值

- 最容易理解。
- 最容易复现。
- 最容易审计。
- 成本可预测。
- 每一步输出都清楚。

### 链式模式的风险

- 灵活性低。
- 上游失败会影响下游。
- 某一步设计错误，整个流程都会偏。
- 不适合动态任务。

## 三种模式对比

| 维度 | 蜂群模式 | Sub 模式 | 链式模式 |
|---|---|---|---|
| 协调方式 | 共享环境、自组织、弱中心 | 主 Agent 拆分和协调 | 固定顺序传递 |
| 任务结构 | 动态任务池 | 明确子任务 | 固定流水线 |
| 灵活性 | 高 | 中 | 低 |
| 可控性 | 中，需要强约束 | 高 | 高 |
| 容错性 | 高 | 中 | 低 |
| 成本可预测性 | 低到中 | 中到高 | 高 |
| 适合任务 | 复杂探索、动态协作 | 多角色分工、可控交付 | 固定流程、标准化处理 |
| 主要风险 | 发散、成本不可控、难收敛 | 主 Agent 拆分不准、子任务重复 | 链条断裂、流程僵化 |

## 如何选择

| 任务特征 | 推荐模式 |
|---|---|
| 任务路径不确定，需要多个 Agent 动态探索 | 蜂群模式 |
| 子任务边界清楚，需要多个子 Agent 分工 | Sub 模式 |
| 流程固定，步骤清楚，不需要动态调整 | 链式模式 |
| 大规模代码审查，需要多维度互相补全 | 蜂群模式 |
| 修复复杂 bug，但问题可以拆成客户端、后端、验证 | Sub 模式 |
| 文档从采集、整理、审查到发布检查 | 链式模式或 Sub 模式 |
| 生产发布检查 | 链式模式 |
| 产品功能开发 | Sub 模式 |

## Heicode 的演进关系

Heicode 可以按以下路径理解：

```text
单 Agent
-> 链式模式：固定流程自动化
-> Sub 模式：主 Agent 拆分多个子 Agent
-> 蜂群模式：多个 Agent 围绕共享任务池自组织协作
```

链式模式强调流程稳定，Sub 模式强调可控分工，蜂群模式强调复杂任务中的群体智能。

真正产品化时，蜂群模式不能脱离权限、预算、审批、密钥和审计。否则 Swarm 的灵活性会变成不可控风险。

## 总结

- 链式模式适合标准流程。
- Sub 模式适合可控分工。
- 蜂群模式适合复杂动态协作。

基于文章理解，蜂群模式是多 Agent 协作的高阶形态；Sub 模式是 Heicode 把多 Agent 协作落到真实交付中的基础能力；链式模式则是最稳定、最容易审计的固定流程。
