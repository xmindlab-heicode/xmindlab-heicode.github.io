# Heicode 产品总览

Heicode 是一个面向软件开发任务的 AI Agent 平台。它把项目上下文、模型接入、工具调用、命令执行、代码修改、测试验证和多 Agent 协作整合到同一套工作流中。

## Heicode 解决什么问题

传统 AI 编程工具通常聚焦单次问答或局部补全。Heicode 更关注完整任务闭环：理解目标、读取仓库、拆分任务、执行修改、运行测试、汇总证据，并把最终决定交给用户。

## 核心组件

| 组件 | 主要作用 |
| --- | --- |
| Heicode Desktop | 本地项目、会话、文件修改、命令审批与结果查看入口 |
| Heicode Cloud | 账号、组织、额度、任务记录和云端控制面 |
| Agent Runtime | 执行单 Agent 开发任务 |
| Agent Swarm | 多 Agent 任务拆分、并行执行、评审与结果聚合 |
| Model Gateway | 连接不同模型提供方并统一请求入口 |
| MCP / Tools | 为 Agent 提供外部工具、数据源和系统能力 |
| Sandbox / Workspace | 为远程任务提供隔离执行环境 |

## 单 Agent 与 Agent Swarm

单 Agent 更适合边界清晰、改动范围较小、需要快速交互的任务。Agent Swarm 更适合复杂、长周期、可以并行拆分的任务。

多 Agent 并不天然优于单 Agent。任务过小、验收标准不清晰或预算不足时，使用 Swarm 可能只会增加成本和协调开销。

## 运行模式

| 模式 | 代码位置 | 执行位置 | 适用场景 |
| --- | --- | --- | --- |
| 本地模式 | 本地设备 | 本地设备 | 私有代码、本地调试、低延迟交互 |
| 云端模式 | 云端工作区 | 远程 Sandbox 或 VM | 长任务、并行任务、持续执行 |
| 混合模式 | 本地与云端 | 本地 + 云端 | 本地控制、远程执行、多设备协作 |

## Heicode 不是什么

Heicode 不能保证所有任务都能自动完成，也不能替代代码审查、测试、权限管理和发布流程。Agent 生成的代码、命令、测试结果和结论都应经过验证。

## 建议阅读顺序

1. 阅读“首次成功路径”，完成第一个可验证任务。
2. 阅读“Agent 权限与安全边界”，确认允许 Agent 执行的操作。
3. 阅读“任务结果与验收”，了解 completed、passed 和 resolved 的区别。
4. 使用 Agent Swarm 前，再阅读多 Agent 的成本、并发和协调机制。
