# Approval, Delivery, and Release

<Callout type="warn" title="Current Status Notice">
  The interface for configuring team-level members, roles, and approval rules is still in the planning/backend stage, and a dedicated team management interface has not yet been made available. The approval requests, short-lived credential derivation, and audit records described on this page are mechanisms that already run in real, single-task/single-Agent scenarios.
</Callout>

When a team uses Heicode, high-risk operations cannot be completed automatically by an Agent by default. Approval is the core mechanism by which a team controls production risk.

## Operations that must be approved

- Production deployment.
- Production database read/write.
- Production secret access.
- Cloud resource creation, deletion, scaling.
- Merging to the main branch.
- Sending sensitive data externally.
- Large-scale model budget consumption.
- Bulk file deletion.

## Approver mechanism

Each high-risk operation can specify a target approval role (`TargetRole`), approved by a member holding that role. Which approval role is assigned to which operation is currently determined by the actual settings at task launch time, not a fixed rule table preset by the platform, nor an approval matrix such as "production deployment requires approval from roles A+B." Teams can assign the appropriate `TargetRole` to different operations based on their own division of labor — for example, prioritizing members familiar with operations and security for production deployment, production database writes, and production secret access, and prioritizing the Owner for large budgets.

## Must be checked before approval

- Operation target.
- Target resource.
- Target environment.
- Scope of impact.
- Risk level.
- Budget impact.
- Whether it is reversible.
- Credential TTL.
- The diff or deployment plan.

## Approval outcomes

Approved:

- Only this specific operation is approved.
- Only valid within the current TTL.
- The platform derives short-lived credentials.
- An audit record is written.

Rejected:

- The high-risk operation will not be executed.
- The task enters a blocked state or an alternative approach.
- The Agent should explain the subsequent options.

## Code review

Code review should check:

- Whether it meets the requirements.
- Whether it only modifies the authorized scope.
- Whether there are unrelated changes.
- Whether there are tests.
- Whether there is error handling.
- Whether there are plaintext secrets.
- Whether it affects production data.

A Reviewer Agent can assist with the review, but the final judgment should be made by the team owner.

## Feature acceptance

At acceptance, check:

- Whether acceptance criteria are met.
- Whether the core flow works.
- Whether edge cases are handled.
- Whether the UI or interface meets requirements.
- Whether tests pass.
- Whether documentation has been updated.

## Pre-release checklist

- The feature has been accepted.
- Tests have passed.
- The code has been reviewed.
- No plaintext secrets.
- The deployment plan has been confirmed.
- The rollback plan has been confirmed.
- Production approval has been completed.
- Monitoring and logging are available.
- Audit records are complete.

## Post-release checklist

- Whether the service is accessible.
- Whether core functionality is normal.
- Whether the error rate is abnormal.
- Whether logs are normal.
- Whether resource consumption is abnormal.
- Whether user feedback is normal.

## Rollback

When a rollback is needed:

1. Stop further deployment.
2. Confirm the last stable version.
3. Execute the rollback.
4. Check the service.
5. Record the reason.
6. Create a fix task.

Rollbacks should also be recorded in the audit trail.
