# Team Management and Security Governance

<Callout type="warn" title="Current Status Notice">
  Organizations, member invitations, role assignment, and the team governance dashboard are still in the planning/backend stage, and there is no dedicated team management interface yet; the parts of administrator responsibilities involving member and role management currently cannot be operated directly through an interface. Secret custody, approval, and auditing mechanisms are capabilities that already run in real, single-task/single-Agent scenarios.
</Callout>

After a team uses Heicode long-term, it needs to periodically govern members, permissions, resources, secrets, approvals, and audits.

## Administrator responsibilities

Administrators are responsible for:

- Inviting and removing members.
- Setting roles.
- Connecting resources.
- Configuring approval rules.
- Managing budgets.
- Viewing audits.
- Handling security incidents.
- Coordinating billing and plan/subscription management.

## Member governance

Check monthly:

- Whether there are departed members.
- Whether there are long-term inactive members.
- Whether there are shared accounts.
- Whether there are members with excessive permissions.
- Whether approvers are still valid.

When a member leaves the team:

- Revoke account permissions.
- Revoke sessions.
- Rotate necessary secrets.
- Transfer tasks.
- Retain audit records.

## Resource governance

Check monthly:

- Which Git repositories are connected.
- Which cloud accounts are connected.
- Which tasks use production resources.
- Which resources have been unused for a long time.
- Which resources have excessive permissions.
- Which resources need to be revoked.

## Secret governance

Requirements:

- Long-lived secrets go into the secret vault.
- The database only stores `secret_ref`.
- Sub-agents do not store long-lived credentials.
- Short-lived credentials have a TTL.
- Secret rotation is recorded.
- Immediately revoke and rotate when a secret is suspected of being leaked.

## Idle accounts

When an account has had no login, no paid usage, no running tasks, no organization activity, or no active subscription activity for 180 consecutive days, Heicode may treat it as an idle account (this policy is subject to the product terms, and the specific automated execution timing may be adjusted).

Accounts that remain unused after notification may have their login restricted, new Agent tasks stopped, model invocation disabled, resource access disconnected, and be entered into the secret vault cleanup process.

## Service suspension or termination

In the presence of illegal activity, violation of a model provider's policies, unauthorized security testing, data exfiltration, malware, credential theft, fraud, abuse, payment failure, chargeback, resale or sharing of accounts/credentials, bypassing approval or security controls, a request from the resource owner or administrator, a legal order, an immediate security risk, or long-term idleness after notification, Heicode may suspend, restrict, or terminate service.

## Handling security incidents

When a security incident is discovered:

1. Stop the related task.
2. Revoke related credentials.
3. Rotate secrets.
4. Check logs and audit records.
5. Notify the relevant owner.
6. Assess the scope of impact.
7. Fix the issue.
8. Hold a retrospective and update the rules.

## Governance metrics

It is recommended that teams track the following metrics themselves (currently these need to be compiled manually from audit records; the platform does not yet have an automatically generated governance dashboard):

- Number of members.
- Active members.
- Idle accounts.
- Number of resource authorizations.
- Number of production approvals.
- Average approval turnaround time.
- Task success rate.
- Number of failed retries.
- Model consumption.
- Number of security incidents.
- Number of secret rotations.
