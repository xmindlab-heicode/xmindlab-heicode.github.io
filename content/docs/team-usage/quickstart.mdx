# Team Quickstart

<Callout type="warn" title="Current Status Notice">
  Team management capabilities such as organizations, member invitations, and role assignment are still in the planning/backend stage, and there is no dedicated team management interface yet. The descriptions on this page related to approvals, secret custody, and auditing are mechanisms that already run in real, single-task/single-Agent scenarios. The parts involving multi-member organizations and team roles are forward-looking usage suggestions.
</Callout>

This page explains how a team should get started the first time it uses Heicode.

## Step 1: Create or join a team

After a team administrator creates an organization, they invite members to join.

Members will receive an invitation, and then:

1. Log in with their own account.
2. Confirm the organization name.
3. Check their own role.
4. Download the Heicode client.
5. View the team's task list.

Do not share the same account across multiple people.

## Step 2: Set up basic roles

Currently there are only 4 real roles in a Heicode organization:

- Owner: responsible for the organization, plan/subscription, billing, and final authority.
- Admin: responsible for members, resources, tasks, and audit.
- Member: an ordinary member participating in tasks.
- Auditor: read-only access to audit records.

A small team may have one person holding multiple roles, but permissions should be clear. Divisions of labor such as technical solutioning, requirement acceptance, deployment/operations, and security policy are currently internal team responsibility agreements, not organization roles that can be selected on the platform.

The Agent role is a separate system, unrelated to the organization roles above. It includes six roles: Product Agent, Architect Agent, Frontend Agent, Backend Agent, Reviewer Agent, and Ops Agent. See [Roles and Permissions](/docs/team-usage/roles-and-permissions) for details.

## Step 3: Connect basic resources

An administrator or owner connects:

- Git repositories.
- Product documentation.
- The SK skill library.
- Cloud accounts.
- Deployment resources.

After connecting, the team should confirm:

- Who can use this resource.
- Which tasks can use this resource.
- Whether it is read-only.
- Whether writes are allowed.
- Whether it is a production resource.
- Which operations require approval.

## Step 4: Set default approval rules

Recommended defaults requiring approval:

- Production deployment.
- Production database read/write.
- Production secret access.
- Cloud resource deletion.
- Merges to the main branch.
- Large-scale model consumption.

Each high-risk operation can specify a target approval role (`TargetRole`), approved by a member holding that role. Which approval role is assigned to which operation is currently determined by the actual settings at task launch time, rather than a fixed rule table preset by the platform. Teams can agree on this based on their own division of labor — for example, prioritizing members familiar with operations and security for approving production deployment, production databases, and production secrets, and prioritizing the Owner for approving large budgets.

## Step 5: Create the team's first task

It is recommended to start with low-risk tasks, such as:

- Generating documentation.
- Fixing a small bug.
- Adding a non-production page.
- Running tests and compiling a report.

Do not let an Agent directly operate on production resources from the very start.

## Step 6: Launch the Agent

The task owner confirms:

- The goal is clear.
- The repository is correct.
- The resource scope is correct.
- The Agent role is reasonable.
- The budget is reasonable.
- High-risk approval rules have been set.

Then launch the Agent.

## Step 7: Team handling of task status

Once a task is launched, the team mainly focuses on:

- Whether input is needed.
- Whether it is waiting for approval.
- Whether it has failed.
- Whether it is ready for acceptance.
- Whether consumption is abnormal.

Not everyone on the team needs to read the full logs. The task owner watches the task status, and relevant roles step in when needed.

## Step 8: Acceptance and retrospective

After completion, check:

- Whether acceptance criteria are met.
- Whether there are test results.
- Whether there has been a code review.
- Whether there is an approval record.
- Whether there is a deployment record.
- Whether there is a cost record.

After the first task is completed, the team should hold a retrospective on:

- Whether the task description was clear.
- Whether resource authorization was appropriate.
- Whether the approver was correct.
- Whether there were too many or too few Agent roles.
- Whether the budget was reasonable.
