# Member Roles and Permissions

<Callout type="warn" title="Current Status Notice">
  Organizations, member invitations, and role assignment are still in the planning/backend stage, and there is no dedicated team management interface yet. The organization roles and member-joining approach described on this page are data models already defined on the backend, which currently cannot be operated directly through an interface. Secret custody, approval, and auditing mechanisms are capabilities that already run in real, single-task/single-Agent scenarios.
</Callout>

The goal of team permissions is to let both members and Agents access only the resources they need.

## Organization roles

Currently there are only 4 real roles in a Heicode organization:

| Role | Main responsibility | Suggested permissions |
|---|---|---|
| Owner | Organization, plan/subscription, billing, final authorization | Global administration, billing, members, resources, audit |
| Admin | Day-to-day management | Members, resources, tasks, audit |
| Member | Ordinary member participating in tasks | Create tasks, use authorized resources, view results |
| Auditor | Read-only access to audit | View audit records |

There are no finer-grained organization roles such as "Tech Lead / Product Lead / Developer / Reviewer / Ops / Security / Billing"; responsibilities such as technical, requirement, operations, and security work can currently only be divided through internal team agreement, not roles that can be selected on the platform.

A planned future role model (not yet launched, for reference only): `Primary Owner / Owner / Admin / Member / Auditor / Custom`, which may support more granular custom roles in the future, but cannot currently be selected or configured.

## Agent roles

The Agent role is a separate system independent from organization roles. There are 6 real Agent roles in the catalog:

| Agent | Suitable for |
|---|---|
| Product Agent | Requirements, user stories, acceptance criteria |
| Architect Agent | Architecture, module boundaries, technical solutions |
| Frontend Agent | Pages, components, interactions |
| Backend Agent | APIs, databases, service logic |
| Reviewer Agent | Code review, risk checks |
| Ops Agent | Build, deployment, monitoring |

## Permission principles

- Each member uses their own account.
- Each member only has the permissions needed to fulfill their responsibilities.
- Each Agent only has the permissions needed for the current task.
- Production resources are not open by default.
- High-risk operations must be approved.
- Permission changes must be recorded.

## Member onboarding

The real member invitation process is simple: select a user and assign them one of the roles Owner / Admin / Member / Auditor. The invitation information includes the member's email, the team they belong to, and their organization role.

Item-by-item permission checkboxes such as "can create tasks / can connect resources / can approve / can view usage / can view audit" are not currently part of the real invitation process — they are only a planned idea for finer-grained permissions that has not yet been implemented.

## Member offboarding

When a member leaves, changes roles, or no longer participates in a project:

1. Remove organization permissions.
2. Revoke sessions.
3. Revoke personal tokens.
4. Transfer tasks they were responsible for.
5. Check their approval permissions.
6. Check resources they have connected to.
7. Rotate secrets if necessary.
8. Retain audit records.

## Permission review

Recommended:

- Check pending approval tasks weekly.
- Check member permissions monthly.
- Check resource authorizations monthly.
- Check model consumption monthly.
- Check secret and audit policies quarterly.

## Risks of shared accounts

Shared accounts lead to:

- Inability to determine who approved a high-risk operation.
- Inability to track who connected a resource.
- Inability to correctly revoke the permissions of a departed member.
- Higher risk of secret leaks.
- Invalidated audit records.

Teams must avoid shared accounts.
