# 成员角色和权限

团队权限的目标是让成员和 Agent 都只访问自己需要的资源。

## 组织角色

| 角色 | 主要职责 | 建议权限 |
|---|---|---|
| Owner | 组织、套餐、账务、最终授权 | 全局管理、账务、成员、资源、审计 |
| Admin | 日常管理 | 成员、资源、任务、审计 |
| Tech Lead | 技术质量 | 任务、资源确认、代码审查 |
| Product Lead | 需求和验收 | 创建任务、确认范围、验收 |
| Developer | 开发协作 | 查看任务、补充需求、查看结果 |
| Reviewer | 审查 | 查看 diff、测试、审计摘要 |
| Ops | 部署和运行 | 云资源、部署审批、运行日志 |
| Security | 安全和合规 | 密钥、权限、审批策略、安全审计 |
| Billing | 费用和套餐 | 用量、账务、发票 |

## Agent 角色

| Agent | 适合做什么 |
|---|---|
| Product Agent | 需求、用户故事、验收标准 |
| Architect Agent | 架构、模块边界、技术方案 |
| Frontend Agent | 页面、组件、交互 |
| Backend Agent | API、数据库、服务逻辑 |
| Reviewer Agent | 代码审查、风险检查 |
| Tester Agent | 测试和回归 |
| Ops Agent | 构建、部署、监控 |
| Security Agent | 密钥、权限、安全风险 |
| Docs Agent | 文档和发布说明 |

## 权限原则

- 每个成员使用自己的账号。
- 每个成员只拥有完成职责所需的权限。
- 每个 Agent 只拥有当前任务所需的权限。
- 生产资源默认不开放。
- 高危操作必须审批。
- 权限变化必须有记录。

## 成员加入

管理员邀请成员时应设置：

- 成员邮箱。
- 所属团队。
- 组织角色。
- 可见项目。
- 是否可创建任务。
- 是否可连接资源。
- 是否可审批。
- 是否可查看用量。
- 是否可查看审计。

## 成员离开

成员离职、转岗或不再参与项目时：

1. 移除组织权限。
2. 撤销会话。
3. 撤销个人 token。
4. 转移其负责的任务。
5. 检查其审批权限。
6. 检查其连接过的资源。
7. 必要时轮换密钥。
8. 保留审计记录。

## 权限复查

建议：

- 每周检查待审批任务。
- 每月检查成员权限。
- 每月检查资源授权。
- 每月检查模型消耗。
- 每季度检查密钥和审计策略。

## 共享账号风险

共享账号会导致：

- 无法判断谁批准了高危操作。
- 无法追踪谁连接了资源。
- 无法正确撤销离职成员权限。
- 容易造成密钥泄漏。
- 审计记录失效。

团队必须避免共享账号。
