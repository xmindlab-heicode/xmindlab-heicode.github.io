# Team Task Collaboration Workflow

The core workflow for a team using Heicode is: the client inputs the goal, the Manager prepares resources, the Agent executes, and the team provides input, approval, and acceptance as needed.

## Where a task starts

Tasks primarily start from the Heicode client.

The Manager does not serve as a web-based coding entry point; it is responsible for:

- Preparing the readiness checklist.
- Resource connections.
- Agent launch.
- Status viewing.
- Usage and audit.

## Task description template

```text
Task name:
Business background:
Target users:
Feature to be completed:
Existing code or documentation:
Target environment:
What must not be done:
Operations requiring approval:
Acceptance criteria:
Deadline:
```

## Team division of labor

Currently there are only four real roles in a Heicode organization: Owner, Admin, Member, and Auditor; the platform does not provide more granular job roles. The table below is a suggested division of responsibilities that a team can reference internally, organized by task phase, rather than a role system enforced by the platform:

| Phase | Suggested owner |
|---|---|
| Requirement input | Members familiar with business requirements |
| Technical boundaries | Members familiar with the technical solution |
| Repository and branches | Members responsible for the code |
| Documentation and SK | Members familiar with product or technical documentation |
| Cloud resources | Members familiar with operations |
| Secrets and approval | Admin or a designated approval role (`TargetRole`) |
| Budget | Owner |
| Acceptance | Requirement owner / technical owner |

## Readiness checklist

The readiness checklist shows:

- Whether Git is connected.
- Whether documentation is connected.
- Whether SK is connected.
- Whether the cloud account is connected.
- Whether cloud resources are selected.
- Whether a budget is set.
- Whether approval rules are set.
- Whether the Agent role is confirmed.

The team should complete the readiness checklist before launching the Agent.

## Pre-launch meeting

For important tasks, it is recommended to hold a short pre-launch meeting to confirm:

- Whether the goal is clear.
- Whether acceptance criteria are explicit.
- Whether the resource scope is safe.
- Whether production is involved.
- Whether the approver is online.
- Whether the budget is sufficient.
- Whether the task needs to be split.

## During task execution

While a task is executing, the team only needs to handle three types of matters:

- Input needed: answer the Agent's questions.
- Waiting for approval: confirm the risk before approving or rejecting.
- Ready for acceptance: check the delivered result.

Do not let all members change the task direction at the same time. It is recommended that the task owner respond in a unified way.

## Splitting tasks

When a task is too large, it should be split.

Split by phase:

```text
Requirement confirmation
-> Architecture solution
-> Frontend implementation
-> Backend implementation
-> Test fixes
-> Deployment and release
```

Split by module:

```text
User system
-> Project system
-> Task system
-> Comment system
-> Notification system
```

Split by risk:

```text
Low-risk code changes
-> Test environment deployment
-> Staging verification
-> Production approval
```

## Task communication norms

Good replies:

```text
Only implement login and the project list.
Do not build the notification module.
Only modify api/projects and app/projects.
Run tests after completion, do not deploy.
```

Bad replies:

```text
Keep going.
Just do everything.
Use your own judgment.
```

The more specific the instruction, the easier it is for the Agent to deliver the correct result.
