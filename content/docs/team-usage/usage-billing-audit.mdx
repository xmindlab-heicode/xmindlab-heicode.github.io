This page explains how a team can view model consumption, budget, billing, resource access records, and audit information.

Heicode brings model invocation, resource access, high-risk approval, and delivery results together into the same task-level view, making it easier for the team to control cost and risk.

## Usage the team should watch

- Total balance.
- Per-task consumption.
- Sub-agent consumption.
- High-cost model invocations.
- Consumption from failed retries.
- Approval for large-scale consumption.

There is currently no independent dashboard for daily/weekly consumption trends — only the cumulative balance and the consumption of individual records can be seen, requiring you to filter raw records by time yourself to estimate trends.

## Budget strategy

| Task type | Suggested budget strategy |
| --- | --- |
| Small fix | Small budget, pause when exceeded |
| Medium feature | Medium budget, alert on long running time |
| MVP build | Phased budget, continue after phase acceptance |
| Production deployment | Small budget, focus on approval |
| Documentation task | Low budget, limit tool invocations |

## Handling abnormal cost

If model consumption is abnormal:

1. Pause the task.
2. Check sub-agent consumption.
3. Check failed retries.
4. Check whether the task is too large.
5. Narrow the scope or split the task.
6. Adjust the model strategy.
7. Set stricter budget thresholds.

## Audit records

Audit events can answer:

```text
Who initiated the operation
Which Agent used the resource
Who approved the high-risk operation (Decision: allow/deny/ask)
When and from which source IP the operation occurred
```

How much quota was consumed needs to be verified separately in the billing records — audit events do not directly carry a quota field. Whether something was deployed to production is also currently not a separately queryable flag — it needs to be judged manually by combining the task description and delivery records.

## Resource access records

The team should be able to view:

- Access time.
- Accessing user.
- Accessing Agent.
- Resource type.
- Permission scope.
- Whether short-lived credentials were used.
- Whether it was approved.
- Operation result.

## Log security

Logs automatically redact common sensitive information, including:

- Plaintext tokens (such as secrets starting with `sk-`).
- `Bearer` authorization headers.
- Secrets in URL parameters.
- Common field names such as `password`/`token`/`access_key`/`private_key` in JSON.

Redaction is based on generic rule matching, not a whitelist that individually recognizes specific names such as "CodeGW key" or "OpenBao token" — if you find that a particular type of secret was not correctly redacted, you should immediately stop the task, notify the security owner, and report it back to the platform so the rules can be extended.

## Monthly retrospective

It is recommended to review monthly:

- Which tasks consumed the most.
- Which tasks failed the most.
- Which approvals waited the longest.
- Which resources were accessed the most.
- Which members have excessive permissions.
- Which secrets need to be rotated.
- Whether the plan/subscription or budget needs adjustment.

The above retrospective currently needs to be compiled manually from raw records; there is no automatically generated monthly report yet.
