# Heicode Agent 蜂群

本页说明 Heicode 中“蜂群模式”是什么、为什么需要它，以及它与单 Agent、Sub 模式的区别。

## 一句话说明

Heicode 的 Agent 蜂群，是一种面向真实软件交付的多 Agent 协作机制。它不是简单地同时启动多个 AI，而是让多个子 Agent 围绕同一个目标，在统一的任务、上下文、资源、产物和审计体系中协同工作。

```text
用户提出目标
-> Heicode 理解需求
-> 系统生成任务图
-> 多个子 Agent 按能力领取任务
-> 执行、交接、验证和修复
-> 高危操作进入审批
-> 产物、日志、用量和审计统一回传
```

## 为什么需要 Agent 蜂群

单个 AI 助手适合回答问题或完成局部任务，但真实软件开发通常不是单点动作。一个产品从想法到上线，往往会涉及：

- 需求理解。
- 产品拆解。
- 技术方案。
- 前端开发。
- 后端开发。
- 测试验证。
- 安全检查。
- 部署上线。
- 日志排查。
- 后续维护。

如果只依赖一个 Agent，常见问题包括：

- 上下文过长后容易遗漏重点。
- 复杂任务拆解不稳定。
- 代码、测试、部署之间缺少交接。
- 失败后不知道如何回流修复。
- 资源权限、密钥和生产操作难以控制。
- 用户无法清楚看到谁做了什么、用了什么资源、结果是什么。

Agent 蜂群的价值，是把一个大目标拆成多个可追踪、可协作、可验证的子任务，并让不同能力的 Agent 在受控边界内共同完成。

## 蜂群模式的核心组成

### 1. Swarm Run

每次启动一个目标，系统都会创建一次 Swarm Run。它是一条完整运行记录，包含：

- 用户目标。
- 会话上下文。
- 可用资源。
- Agent 编队。
- 动态任务图。
- 运行状态。
- 审批策略。
- 预算限制。
- 产物索引。
- 日志和审计。

### 2. 动态任务图

Heicode 不把任务固定成一条死流程，而是根据目标和上下文生成可变化的任务图。

例如：

```text
理解需求
-> 读取相关代码
-> 设计接口和数据结构
-> 修改后端代码
-> 修改前端页面
-> 运行测试
-> 修复失败
-> 生成交付说明
```

如果执行过程中发现新问题，系统可以继续生成修复、审批或补充确认任务。

### 3. 能力编队

系统不会固定每次都启动同一组 Agent，而是根据任务目标动态组织能力。常见角色包括：

- Product Agent
- Architect Agent
- Frontend Agent
- Backend Agent
- Reviewer Agent
- Test Agent
- Ops Agent
- Docs Agent

### 4. 共同资源层

共同资源层是蜂群协作的共享工作台。没有它，多个 Agent 只是同时运行；有了它，多个 Agent 才能真正协作。

| 层面 | 作用 |
| --- | --- |
| 任务层 | 管理任务池、任务状态、领取、交接和失败回流 |
| 上下文层 | 管理需求、代码范围、历史失败和决策记录 |
| 评分层 | 判断任务优先级、能力匹配度、风险和预算影响 |
| 产物层 | 保存代码 diff、测试报告、文档和部署计划 |
| 权限层 | 管理 Git、文档、SK、云资源和密钥引用 |
| 审批层 | 管理生产部署、云操作、密钥访问等高危确认 |
| 审计层 | 记录每个动作的主体、资源、时间、结果和用量 |

### 5. 受控自治

Heicode 蜂群不是完全放任 AI 自动操作，而是允许自动协作，但不能越过权限、密钥、预算、审批和审计边界。

## 和 Sub 模式有什么区别

| 维度 | 蜂群模式 | Sub 模式 |
| --- | --- | --- |
| 协作方式 | 围绕共享任务池自组织协作 | 主 Agent 拆分和汇总 |
| 任务结构 | 动态变化 | 边界较清楚 |
| 共享状态 | 强依赖 | 相对较弱 |
| 可控性 | 较低，需要更强约束 | 高 |
| 成本 | 通常更高 | 通常更可控 |

## 蜂群模式适合什么场景

更适合：

- 复杂产品功能探索。
- 大范围代码审查和问题定位。
- 复杂故障排查。
- 多角色并发研究和交叉验证。
- 需要多个 Agent 持续互相补位的任务。

不适合：

- 单页文档修改。
- 很小的 bug 修复。
- 必须严格线性执行的任务。
- 预算极低的任务。

## 蜂群模式需要哪些控制

蜂群模式能力更强，但必须额外控制：

- 最大 Agent 数量。
- 最大预算。
- 最大运行轮次。
- 高危审批规则。
- 任务停止条件。
- 冲突处理方式。
- 共享状态和事件记录。

## 使用建议

- 先从 Sub 模式验证任务是否适合拆分。
- 确认需要动态协作后，再启用蜂群模式。
- 涉及生产环境时，一定保留人工审批。
- 预算较高的任务要拆阶段执行，不要一次性放开。
