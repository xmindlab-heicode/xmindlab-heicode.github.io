\--- title: 选择 Agnet 工作模式 description: 说明 Sub 模式、蜂群模式、瀑布模式和敏捷模式的差异、适用场景、配置方式、监控指标和切换规则。 updated: 2026-05-14 \--- # 选择 Agnet 工作模式 Agnet 启动前需要选择工作模式。工作模式决定任务如何拆分、子 Agnet 如何协作、什么时候需要你确认、怎样验收交付结果。 Heicode 的工作模式分为两类： \- 协作模式：决定多个 Agnet 如何一起工作，包括 Sub 模式和蜂群模式。 \- 交付模式：决定任务如何推进和验收，包括瀑布模式和敏捷模式。 Sub 模式不是蜂群模式。Sub 模式是子 Agnet 拆分和显式协作的基础能力；蜂群模式是正在完成的高阶多 Agnet 协作形态，借鉴了 Swarm 的共享环境、自组织、隐式协作和涌现结果。 ## 四种模式快速对比 | 维度 | Sub 模式 | 蜂群模式 | 瀑布模式 | 敏捷模式 | |---|---|---|---|---| | 类型 | 协作模式 | 协作模式 | 交付模式 | 交付模式 | | 核心思想 | 主 Agnet 拆分任务，子 Agnet 按职责执行 | 多个 Agnet 围绕共享环境自组织协作 | 按固定阶段顺序推进 | 按短周期持续迭代 | | 协调方式 | 主 Agnet 显式分配、交接、聚合 | 共享任务池、状态事件、局部决策、收敛判断 | 上一阶段确认后进入下一阶段 | 每轮计划、执行、验收、调整 | | 用户参与 | 关键问题、审批和验收时介入 | 设定边界、处理异常、验收聚合结果 | 每个阶段结束时确认 | 每个迭代结束时反馈 | | 适合任务 | 明确分工、多模块开发、复杂排障 | 多源研究、复杂探索、大规模并行审查、动态任务 | 生产发布、合规文档、迁移、正式交付 | 产品功能、MVP、UI/体验优化 | | 主要风险 | 主 Agnet 拆分不准，子任务重复 | 协作发散、收敛慢、成本不可控 | 前期需求错误会影响后续 | 频繁变更可能扩大范围 | | 预算控制 | 按子 Agnet 控制 | 按任务池、迭代轮次和收敛阈值控制 | 按阶段控制 | 按迭代控制 | | 验收方式 | 主 Agnet 聚合后验收 | 收敛后聚合验收，必要时人工裁决 | 阶段验收和最终验收 | 每轮增量验收 | ## Sub 模式 Sub 模式是 Heicode 的基础多 Agnet 协作方式。它的重点不是“蜂群自组织”，而是“主任务可控拆分”。 在 Sub 模式中，主 Agnet 负责理解用户目标、拆分子任务、分配给子 Agnet、汇总结果并对用户交付。子 Agnet 负责单一职责，例如研究、前端、后端、测试、审查或部署。 ### Sub 模式如何工作 ```text 用户目标 -> 主 Agnet 理解目标 -> 主 Agnet 拆分子任务 -> 创建或唤起子 Agnet -> 子 Agnet 执行自己的任务 -> 子 Agnet 显式交接结果 -> 主 Agnet 聚合 -> 验证和交付 ``` ### Sub 模式讲解要点 | 概念 | 含义 | |---|---| | 主 Agnet | 中央协调者，负责拆分、分配、聚合和汇报 | | 子 Agnet | 单一职责执行者，只处理被分配的局部任务 | | 显式任务 | 子 Agnet 的任务由主 Agnet 或用户明确给出 | | Handoff | 子 Agnet 完成后把结果或问题交给主 Agnet 或下一个子 Agnet | | 聚合 | 主 Agnet 汇总各子任务，形成统一交付 | | 可控边界 | 每个子 Agnet 有明确资源、权限、预算和审批规则 | ### Sub 模式和文章中 Swarm 的关系 掘金文章讲的 Swarm 强调“个体简单、群体智能”：多个 Agent 通过共享环境、局部信息、状态变化和间接通信形成整体结果。 Sub 模式可以用这篇文章来帮助客户理解“为什么需要多个子 Agnet”，但它不是完整 Swarm： \- Sub 模式有主 Agnet，Swarm 更强调去中心化或弱中心化。 \- Sub 模式的任务通常由主 Agnet 显式分配，Swarm 更强调共享任务池和自组织领取。 \- Sub 模式以交接和聚合为主，Swarm 还需要收敛判断、信息素式状态积累和动态协作。 \- Sub 模式适合稳定交付，Swarm 更适合复杂探索和动态问题。 简单说：Sub 模式是蜂群模式的基础材料，不是蜂群模式本身。 ### 什么时候选择 Sub 模式 选择 Sub 模式的情况： \- 一个任务需要多个专业角色参与。 \- 子任务边界比较清楚。 \- 需要主 Agnet 统一控制方向。 \- 需要代码实现、测试、审查分开处理。 \- 问题复杂，但还不需要完全自组织协作。 \- 需要把任务控制在明确预算和权限内。 不建议选择 Sub 模式的情况： \- 任务很小，单个 Agnet 可以完成。 \- 需求不清楚，连目标都没有定义。 \- 子任务之间强依赖共享状态和动态调整。 \- 需要大量 Agent 自动寻找任务和相互影响。 \- 输出必须完全按固定阶段签收。 ### Sub 模式配置建议 推荐子 Agnet 数量： | 任务复杂度 | 建议数量 | |---|---| | 小任务 | 1 到 2 个 | | 中等任务 | 3 到 5 个 | | 大任务 | 5 到 8 个 | 常见角色组合： | 场景 | 推荐组合 | |---|---| | 新功能 | Product、Frontend、Backend、Reviewer | | 复杂 bug | Debug、Backend、Test、Reviewer | | 文档和资源中心 | Product、Docs、Reviewer | | 云部署 | Ops、Security、Reviewer | | 大型重构 | Architect、Backend、Test、Reviewer | 权限建议： \- 每个子 Agnet 只拿自己需要的资源。 \- Reviewer Agnet 默认不负责直接修改生产资源。 \- Ops Agnet 可以准备部署，但生产部署必须审批。 \- Security Agnet 可以审查密钥和权限，但不应长期持有明文密钥。 \- 子 Agnet 之间共享的是任务状态和结果，不共享不必要的密钥。 ## 蜂群模式 蜂群模式是 Heicode 正在完成的高阶多 Agnet 协作模式。它基于 Swarm 思路：多个 Agnet 不只是被主 Agnet 分配任务，而是围绕共享任务池、共享状态和局部反馈协作，最终通过收敛判断形成整体结果。 蜂群模式适合复杂、动态、边界不断变化的任务。它不是简单“多开几个子 Agnet”，而是一套任务环境和协作规则。 ### 蜂群模式如何工作 ```text 用户目标 -> 创建共享任务环境 -> 初始化多个 Agnet 角色 -> Agnet 感知任务池和共享状态 -> Agnet 自主选择或竞争子任务 -> 执行、广播状态、更新共享结果 -> 发生 Handoff 或协作补全 -> 平台判断是否收敛 -> 聚合、验证和交付 ``` ### 蜂群模式的核心组件 | 组件 | 作用 | |---|---| | Agent Pool | 管理多个 Agnet 实例和角色能力 | | 共享任务池 | 保存待处理、进行中、阻塞、完成和失败的任务 | | 共享状态 | 保存关键事实、约束、阶段结论和冲突点 | | 事件总线 | 记录状态变化，让相关 Agnet 感知进展 | | Handoff 机制 | 在角色之间交接上下文、结果和控制权 | | 收敛判断 | 判断任务是否达到质量、完整性或停止条件 | | 结果聚合器 | 合并多个 Agnet 的结果，去重、排序、统一格式 | | 监控和审计 | 记录用量、权限、异常、审批和交付证据 | ### 蜂群模式和 Sub 模式的差异 | 维度 | Sub 模式 | 蜂群模式 | |---|---|---| | 控制方式 | 主 Agnet 明确分配任务 | 多 Agnet 围绕任务环境自组织 | | 任务来源 | 主 Agnet 拆分后下发 | 任务池持续生成、领取、更新 | | 协作方式 | 显式交接为主 | 状态共享、事件驱动、交接并存 | | 失败处理 | 主 Agnet 重新分配或停止 | 其他 Agnet 可接替、补全或绕行 | | 结果形成 | 主 Agnet 汇总 | 聚合器基于多 Agent 结果收敛 | | 适用规模 | 小到中型多角色任务 | 中到大型复杂动态任务 | | 风险控制 | 容易控制 | 必须依赖预算、收敛阈值和审批边界 | ### 什么时候选择蜂群模式 选择蜂群模式的情况： \- 任务需要多个 Agnet 从不同方向同时探索。 \- 子任务之间存在动态依赖，无法一次性拆清楚。 \- 需要持续吸收新信息并调整路径。 \- 需要多个结果互相校验、补全或竞争。 \- 需要处理大规模代码审查、资料研究、复杂故障定位。 \- 单个主 Agnet 容易成为瓶颈。 不建议选择蜂群模式的情况： \- 任务目标很小。 \- 验收标准不清楚。 \- 预算很紧。 \- 没有人处理异常审批。 \- 涉及生产高危操作但没有明确边界。 \- 需要强一致事务或严格线性流程。 ### 蜂群模式配置建议 启动前确认： \- 共享任务池的初始任务。 \- 每类 Agnet 可以领取什么任务。 \- 每类 Agnet 可以访问哪些资源。 \- 任务收敛条件是什么。 \- 最大轮次、最大时间和最大预算。 \- 出现冲突时由谁裁决。 \- 哪些动作必须人工审批。 推荐约束： \- 设置最大 Agnet 数量。 \- 设置单个 Agnet 最大连续运行时间。 \- 设置任务池最大未决任务数。 \- 设置重复失败自动降级或暂停。 \- 设置收敛阈值，例如测试通过、Reviewer 通过、文档覆盖完整。 \- 生产操作必须退出蜂群自动流程，进入审批或瀑布发布阶段。 ### 蜂群模式监控重点 重点观察： \- 任务池是否不断膨胀。 \- 是否有多个 Agnet 重复做同一件事。 \- 是否出现循环交接。 \- 是否有共享状态冲突。 \- 是否迟迟无法收敛。 \- 哪些 Agnet 消耗最多预算。 \- 是否有高危权限请求。 \- 聚合结果是否有明确证据。 发现问题时可以： \- 冻结任务池新增。 \- 停止某个异常 Agnet。 \- 降低并发数量。 \- 强制主 Agnet 汇总阶段结果。 \- 切换到 Sub 模式做可控拆分。 \- 切换到瀑布模式处理正式交付。 ## 瀑布模式 瀑布模式适合目标清楚、阶段顺序明确、风险需要严格控制的任务。它要求一个阶段完成并确认后，才进入下一个阶段。 ### 瀑布模式如何工作 ```text 需求确认 -> 方案设计 -> 实现 -> 测试 -> 审查 -> 发布准备 -> 发布审批 -> 交付归档 ``` 每个阶段都应该有明确输出。 | 阶段 | 典型输出 | |---|---| | 需求确认 | 需求摘要、范围边界、验收标准 | | 方案设计 | 技术方案、风险点、资源计划 | | 实现 | 代码 diff、文档改动、配置变更 | | 测试 | 测试结果、失败项、修复记录 | | 审查 | 审查结论、风险说明、是否可发布 | | 发布准备 | 构建记录、部署计划、回滚方案 | | 发布审批 | 审批记录、凭证 TTL、影响范围 | | 交付归档 | 交付说明、审计摘要、后续事项 | ### 什么时候选择瀑布模式 选择瀑布模式的情况： \- 需求已经明确。 \- 交付边界固定。 \- 涉及生产发布。 \- 涉及客户交付、合规、法律或安全内容。 \- 需要完整审计链路。 \- 团队希望每个阶段都可签字确认。 不建议选择瀑布模式的情况： \- 目标还在探索。 \- 产品方向需要频繁调整。 \- 需要快速试错多个方案。 \- 不知道最终交付应该是什么。 ## 敏捷模式 敏捷模式适合需要持续迭代的任务。它不追求一次性把所有内容做完，而是每一轮交付一个可验证的增量。 ### 敏捷模式如何工作 ```text 建立目标 -> 选择本轮范围 -> 执行 -> 验证 -> 演示或交付 -> 收集反馈 -> 调整下一轮 ``` 每一轮都应回答三个问题： \- 本轮做什么。 \- 本轮不做什么。 \- 本轮怎样算完成。 ### 什么时候选择敏捷模式 选择敏捷模式的情况： \- 产品需求还会变化。 \- 希望先完成 MVP。 \- UI、交互、文案需要多次调整。 \- 需要客户或团队每轮反馈。 \- 任务可以拆成多个小增量。 \- 不希望一次性开放太大权限。 不建议选择敏捷模式的情况： \- 一次性迁移或发布，不能分多轮。 \- 需求已经严格冻结。 \- 每一轮反馈都无人负责。 \- 团队没有时间验收增量。 ## 如何选择模式 可以按以下规则选择： | 任务特征 | 推荐模式 | |---|---| | 多角色分工清楚，需要主 Agnet 控制 | Sub 模式 | | 多方向探索，任务会动态生成和互相影响 | 蜂群模式 | | 需求明确、风险较高、需要阶段签收 | 瀑布模式 | | 需求会变化、需要持续反馈 | 敏捷模式 | | 先探索方案，再进入稳定交付 | 蜂群模式或 Sub 模式，然后切换瀑布模式 | | 先做 MVP，再不断优化 | 敏捷模式 | | 涉及生产部署和回滚 | 瀑布模式 | | 需要多个 Agnet 分别审查安全、性能、测试并互相补全 | 蜂群模式 | ## 模式可以组合吗 可以。协作模式和交付模式通常组合使用。 常见组合： \- 敏捷加 Sub：每轮迭代中，由多个子 Agnet 分工完成本轮目标。 \- 敏捷加蜂群：每轮迭代中，用蜂群快速探索方案、风险和实现路径。 \- 瀑布加 Sub：每个阶段内用子 Agnet 分工，阶段之间严格验收。 \- 蜂群到瀑布：探索和收敛完成后，进入正式发布阶段。 \- Sub 到蜂群：发现静态拆分不足，需要任务池和自组织协作。 切换前应确认： \- 当前交付物是否已保存。 \- 未完成任务是否已归档。 \- 预算是否重新设置。 \- 权限是否需要收缩或扩大。 \- 审批规则是否仍然适用。 \- 审计中是否记录切换原因。 ## 模式选择示例 ### 示例一：修复登录失败 推荐：Sub 模式。 原因： \- Debug Agnet 分析报错。 \- Backend Agnet 检查接口。 \- Client Agnet 检查回跳。 \- Reviewer Agnet 验证修复。 \- 子任务边界清楚，不需要完整蜂群自组织。 ### 示例二：大规模代码审查 推荐：蜂群模式。 原因： \- 安全、性能、测试、依赖、架构可以并行审查。 \- 问题会动态生成后续检查任务。 \- 多个 Agnet 可以互相补全和校验。 \- 最终需要聚合器统一输出风险清单。 ### 示例三：上线法律条款页面 推荐：瀑布模式。 原因： \- 内容需要准确。 \- 发布前需要审查。 \- 生产部署需要审批。 \- 交付后需要归档审计。 ### 示例四：设计资源中心 推荐：敏捷模式。 原因： \- 内容和结构需要多轮确认。 \- 可以先做文档版，再做站内页面。 \- 每一轮都能产出可检查内容。 ## 基于文章理解蜂群模式 掘金文章中的 Swarm 思想可以帮助理解 Heicode 蜂群模式： \- 单个 Agent 能力有限，多个 Agent 可以形成群体智能。 \- Agent 通过局部信息和共享环境协作，而不是每一步都等中央调度。 \- 共享状态类似“信息素”，让后续 Agent 感知已有发现和任务价值。 \- Handoff 让任务在角色之间流转。 \- 收敛判断决定什么时候停止探索并输出结果。 Heicode 在产品化时会增加平台约束： \- 任务负责人。 \- 权限边界。 \- 审批规则。 \- 预算上限。 \- 凭证 TTL。 \- 审计记录。 \- 停止和接管能力。 这意味着客户获得的是可控蜂群，而不是不可追踪的自动化扩散。 ##
