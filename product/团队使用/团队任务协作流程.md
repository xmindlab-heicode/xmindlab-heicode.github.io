\--- title: 团队任务协作流程 description: 团队如何创建任务、补充资源、启动 Agent、协作处理状态和拆分任务。 updated: 2026-05-14 \--- # 团队任务协作流程 团队使用 Heicode 的核心工作流是：客户端输入目标，Manager 准备资源，Agent 执行，团队在需要时输入、审批和验收。 ## 任务从哪里开始 任务主要从 Heicode 客户端开始。 Manager 不作为网页编码入口，它负责： \- 准备清单。 \- 资源连接。 \- Agent 启动。 \- 状态查看。 \- 用量和审计。 ## 任务描述模板 ```text 任务名称： 业务背景： 目标用户： 要完成的功能： 已有代码或文档： 目标环境： 不能做什么： 必须审批的操作： 验收标准： 截止时间： ``` ## 团队分工 | 阶段 | 负责人 | |---|---| | 需求输入 | Product Lead | | 技术边界 | Tech Lead | | 仓库和分支 | Tech Lead / Developer | | 文档和 SK | Product Lead / Tech Lead | | 云资源 | Ops | | 密钥和审批 | Security / Admin | | 预算 | Owner / Billing | | 验收 | Product Lead / Tech Lead | ## 准备清单 准备清单会显示： \- Git 是否连接。 \- 文档是否连接。 \- SK 是否连接。 \- 云账号是否连接。 \- 云资源是否选择。 \- 预算是否设置。 \- 审批规则是否设置。 \- Agent 角色是否确认。 团队应先补齐准备清单，再启动 Agent。 ## 启动前会议 重要任务建议启动前短会确认： \- 目标是否清楚。 \- 验收标准是否明确。 \- 资源范围是否安全。 \- 是否涉及生产。 \- 审批人是否在线。 \- 预算是否足够。 \- 是否需要拆分任务。 ## 任务执行中 任务执行中，团队只需要处理三类事项： \- 需要输入：回答 Agent 的问题。 \- 等待审批：确认风险后批准或拒绝。 \- 可验收：检查交付结果。 不要让所有成员都同时修改任务方向。建议由任务负责人统一回复。 ## 任务拆分 任务过大时应拆分。 按阶段拆： ```text 需求确认 -> 架构方案 -> 前端实现 -> 后端实现 -> 测试修复 -> 部署上线 ``` 按模块拆： ```text 用户系统 -> 项目系统 -> 任务系统 -> 评论系统 -> 通知系统 ``` 按风险拆： ```text 低风险代码修改 -> 测试环境部署 -> 预发验证 -> 生产审批 ``` ## 任务沟通规范 好的回复： ```text 只实现登录和项目列表。 不要做通知模块。 只改 api/projects 和 app/projects。 完成后运行测试，不要部署。 ``` 不好的回复： ```text 继续。 都做一下。 你自己判断。 ``` 越具体，Agent 越容易交付正确结果。
